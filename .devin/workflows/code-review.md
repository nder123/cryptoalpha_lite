# Цель
Снизить регрессии: каждый PR понятен, проверяем, тестируем.

# Требования к PR
- Короткое описание: что изменили и зачем.
- Ссылка на логи/скрин/пример команды проверки (если применимо).
- Не смешивать несвязанные изменения.

# Минимальные гейты
- CI зелёный.
- Минимум 1 одобрение (для критичных изменений — 2).

# Что проверяет ревьюер
- Архитектура: нет ли скрытых побочных эффектов.
- Риски: что будет в рантайме, что будет после рестарта.
- Тестируемость: есть ли тесты на ключевую логику.
- Логи/наблюдаемость: можно ли понять, что система делает.

# После мержа
- Проверить основную метрику/логи в течение 30–60 минут (минимум).
