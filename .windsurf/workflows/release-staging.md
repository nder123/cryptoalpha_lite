# Цель
Выпускать изменения без сюрпризов: сначала изолированно, затем постепенно.

# 1) Staging деплой
- Развернуть в отдельной среде (отдельные Redis/DB/ключи).
- Использовать тестовые/анонимные данные.
- Прогнать smoke:
  - `/api/health`
  - ключевые эндпоинты (в т.ч. `/api/rl/status`).

# 2) Интеграционные и e2e в staging
- Автоматические сценарии на критичные флоу.
- Ручная проверка:
  - запуск сервисов
  - основные экраны/дашборд (если есть)
  - события в логах

# 3) Нагрузочное тестирование
- Проверить:
  - latency
  - error rate
  - CPU/RAM
  - Redis/DB нагрузку
- Зафиксировать базовые цифры «до/после».

# 4) Релиз: canary или blue/green
- Canary:
  - включить на небольшой доле/одной ноде
  - мониторинг 30–60 минут
- Blue/green:
  - поднять green
  - переключить трафик

# 5) Мониторинг и откат
- Мониторить:
  - ошибки (5xx)
  - ключевые события
  - задержки
- Откат:
  - вернуть предыдущую версию
  - проверить, что система стабилизировалась

# 6) Пост-релиз и ретро
- Зафиксировать:
  - что сломалось/что улучшить
  - какие тесты добавить
  - какие алерты нужны
